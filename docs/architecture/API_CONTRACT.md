# Battery Brain -- API Contract

This document defines the REST and WebSocket API contract between the Battery Brain backend (Python/FastAPI) and its clients (web dashboard, mobile app).

Base URL: `http://localhost:3000/api/v1`
WebSocket: `ws://localhost:3000/ws`
OpenAPI spec: `http://localhost:3000/openapi.json` (auto-generated by FastAPI from Pydantic models)

**Note on types:** The backend defines all request/response shapes as Pydantic v2 models. FastAPI auto-generates an OpenAPI 3.1 spec from these models. Frontend TypeScript types are derived from this spec using `openapi-typescript` -- see the "Type Sharing" section at the end of this document.

## Authentication

All endpoints require an `Authorization: Bearer <api-key>` header. The API key is configured in the backend environment. For WebSocket, the token is passed as a query parameter: `ws://localhost:3000/ws?token=<api-key>`.

#### `GET /pricing/current`

Returns the current electricity price and forecast.

**Response 200:**
```json
{
  "current": {
    "per_kwh": 8.5,
    "feed_in_per_kwh": 3.2,
    "descriptor": "low",
    "renewables_pct": 45,
    "spike_status": "none",
    "updated_at": "2026-03-25T10:05:00+11:00"
  },
  "forecast": [
    {
      "start_time": "2026-03-25T10:00:00+11:00",
      "end_time": "2026-03-25T10:30:00+11:00",
      "per_kwh": 8.5,
      "descriptor": "low",
      "renewables_pct": 45
    }
  ]
}
```

#### `GET /pricing/history?from={iso8601}&to={iso8601}&interval={5m|30m|1h|1d}`

Returns historical price data for charts and analytics.

**Query Parameters:**
| Param | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| from | ISO 8601 | Yes | -- | Start of time range |
| to | ISO 8601 | No | now | End of time range |
| interval | string | No | 30m | Aggregation interval |

**Response 200:**
```json
{
  "interval": "30m",
  "data": [
    {
      "time": "2026-03-25T10:00:00+11:00",
      "avg_per_kwh": 8.5,
      "min_per_kwh": 7.2,
      "max_per_kwh": 9.8,
      "avg_feed_in": 3.2
    }
  ]
}
```

### Schedule

#### `GET /schedule`

Returns the current charge/discharge schedule.

**Response 200:**
```json
{
  "generated_at": "2026-03-25T10:00:00+11:00",
  "slots": [
    {
      "start_time": "2026-03-25T10:00:00+11:00",
      "end_time": "2026-03-25T10:30:00+11:00",
      "action": "CHARGE",
      "reason": "Low price period (8.5c/kWh), battery at 72%",
      "estimated_price": 8.5,
      "estimated_solar_w": 3200,
      "profile_id": "prof_default",
      "profile_name": "Balanced"
    }
  ],
  "estimated_savings_today": 4.52
}
```

#### `POST /schedule/override`

Manually override the current schedule action. The override lasts until the specified end time or until cancelled.

**Request:**
```json
{
  "action": "DISCHARGE",
  "end_time": "2026-03-25T12:00:00+11:00",
  "reason": "Manual override - expecting guests"
}
```

`action` must be one of: `CHARGE`, `HOLD`, `DISCHARGE`, `AUTO`.

**Response 200:**
```json
{
  "override_id": "ovr_abc123",
  "action": "DISCHARGE",
  "started_at": "2026-03-25T10:06:00+11:00",
  "ends_at": "2026-03-25T12:00:00+11:00",
  "status": "active"
}
```

#### `DELETE /schedule/override`

Cancel the active manual override and return to the computed schedule.

**Response 200:**
```json
{
  "status": "cancelled",
  "resumed_action": "HOLD"
}
```

### Preferences

#### `GET /preferences`

Returns user preferences. Aggressiveness and scheduling are managed via profiles and calendar rules (see below), not preferences.

**Response 200:**
```json
{
  "min_soc": 20,
  "auto_mode_enabled": true,
  "notifications": {
    "price_spike": true,
    "battery_low": true,
    "schedule_change": false,
    "daily_summary": true
  }
}
```

#### `PATCH /preferences`

Update one or more preferences. Only include fields to change.

**Request:**
```json
{
  "min_soc": 25,
  "notifications": {
    "price_spike": false
  }
}
```

**Response 200:** Returns the full updated preferences object (same shape as `GET /preferences`).

### Profiles (Aggressiveness)

Profiles control how aggressively the decision engine operates across three dimensions. Each parameter is a float from 0.0 (conservative) to 1.0 (aggressive). One profile must be marked as the default.

#### `GET /profiles`

Returns all profiles.

**Response 200:**
```json
{
  "profiles": [
    {
      "id": "prof_default",
      "name": "Balanced",
      "export_aggressiveness": 0.5,
      "preservation_aggressiveness": 0.5,
      "import_aggressiveness": 0.5,
      "is_default": true,
      "created_at": "2026-03-25T10:00:00Z",
      "updated_at": "2026-03-25T10:00:00Z"
    },
    {
      "id": "prof_peak_export",
      "name": "Peak Export",
      "export_aggressiveness": 0.9,
      "preservation_aggressiveness": 0.2,
      "import_aggressiveness": 0.7,
      "is_default": false,
      "created_at": "2026-03-25T10:00:00Z",
      "updated_at": "2026-03-25T10:00:00Z"
    }
  ]
}
```

#### `GET /profiles/{id}`

Returns a single profile.

**Response 200:** Single profile object (same shape as array element above).

#### `POST /profiles`

Create a new profile.

**Request:**
```json
{
  "name": "Weekend Chill",
  "export_aggressiveness": 0.3,
  "preservation_aggressiveness": 0.7,
  "import_aggressiveness": 0.3
}
```

**Response 201:** Returns the created profile with generated `id`.

#### `PATCH /profiles/{id}`

Update a profile. Only include fields to change.

**Request:**
```json
{
  "export_aggressiveness": 0.8
}
```

**Response 200:** Returns the full updated profile.

#### `DELETE /profiles/{id}`

Delete a profile. Fails if the profile is the default or has active calendar rules.

**Response 204:** No content.
**Response 409:** Profile is the default or has active calendar rules.

#### `POST /profiles/{id}/set-default`

Set a profile as the default. The previous default is unset.

**Response 200:** Returns the updated profile with `is_default: true`.

### Calendar Rules (Recurring Schedules)

Calendar rules activate a profile during recurring time periods (e.g., every weekday 4-8pm).

#### `GET /calendar/rules`

Returns all recurring calendar rules.

**Response 200:**
```json
{
  "rules": [
    {
      "id": "rule_peak_export",
      "profile_id": "prof_peak_export",
      "profile_name": "Peak Export",
      "name": "Weekday evening peak",
      "days_of_week": [0, 1, 2, 3, 4],
      "start_time": "16:00",
      "end_time": "20:00",
      "priority": 10,
      "enabled": true,
      "created_at": "2026-03-25T10:00:00Z"
    }
  ]
}
```

`days_of_week`: Array of integers. 0=Monday, 6=Sunday.

`priority`: Higher value wins when multiple rules overlap. Default: 0.

#### `POST /calendar/rules`

Create a recurring calendar rule.

**Request:**
```json
{
  "profile_id": "prof_peak_export",
  "name": "Weekday evening peak",
  "days_of_week": [0, 1, 2, 3, 4],
  "start_time": "16:00",
  "end_time": "20:00",
  "priority": 10
}
```

**Response 201:** Returns the created rule with generated `id`.

#### `PATCH /calendar/rules/{id}`

Update a calendar rule. Only include fields to change.

**Response 200:** Returns the full updated rule.

#### `DELETE /calendar/rules/{id}`

Delete a calendar rule.

**Response 204:** No content.

### Calendar Overrides (One-Off)

One-off overrides activate a profile for a specific date/time range. They take priority over recurring rules.

#### `GET /calendar/overrides`

Returns all one-off overrides (optionally filtered by date range).

**Query Parameters:**
| Param | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| from | ISO 8601 | No | now | Start of range |
| to | ISO 8601 | No | +7 days | End of range |

**Response 200:**
```json
{
  "overrides": [
    {
      "id": "ovr_tues_preserve",
      "profile_id": "prof_preserve",
      "profile_name": "Preserve Battery",
      "name": "Guests visiting Tuesday",
      "start_datetime": "2026-03-31T00:00:00+11:00",
      "end_datetime": "2026-03-31T23:59:59+11:00",
      "created_at": "2026-03-25T10:00:00Z"
    }
  ]
}
```

#### `POST /calendar/overrides`

Create a one-off override.

**Request:**
```json
{
  "profile_id": "prof_preserve",
  "name": "Guests visiting Tuesday",
  "start_datetime": "2026-03-31T00:00:00+11:00",
  "end_datetime": "2026-03-31T23:59:59+11:00"
}
```

**Response 201:** Returns the created override with generated `id`.

#### `DELETE /calendar/overrides/{id}`

Delete a one-off override.

**Response 204:** No content.

## WebSocket Events

Connect to `ws://localhost:3000/ws?token=<api-key>`. The server sends JSON messages with a `type` field.

### Server -> Client Events

#### `battery.update`

Sent on each telemetry poll (~3 minutes by default, adaptive based on API budget and price events). See ADR-008 for rate limiting details.

```json
{
  "type": "battery.update",
  "data": {
    "soc": 73,
    "power_w": 1500,
    "mode": "charging",
    "solar_w": 3200,
    "load_w": 1200,
    "grid_w": -500,
    "temperature": 28.5,
    "timestamp": "2026-03-25T10:06:00+11:00"
  }
}
```

#### `price.update`

Sent when price data changes (approximately every 5 minutes).

```json
{
  "type": "price.update",
  "data": {
    "current_per_kwh": 12.3,
    "feed_in_per_kwh": 3.5,
    "descriptor": "neutral",
    "renewables_pct": 38,
    "timestamp": "2026-03-25T10:10:00+11:00"
  }
}
```

#### `price.spike`

Sent when a price spike is detected or imminent.

```json
{
  "type": "price.spike",
  "data": {
    "current_per_kwh": 85.0,
    "descriptor": "spike",
    "expected_duration_minutes": 30,
    "action_taken": "DISCHARGE",
    "timestamp": "2026-03-25T17:30:00+11:00"
  }
}
```

#### `schedule.update`

Sent when the decision engine produces a new schedule or when a manual override is applied/cancelled.

```json
{
  "type": "schedule.update",
  "data": {
    "current_action": "DISCHARGE",
    "is_override": false,
    "next_change_at": "2026-03-25T18:00:00+11:00",
    "next_action": "HOLD",
    "timestamp": "2026-03-25T17:30:00+11:00"
  }
}
```

#### `profile.change`

Sent when the active aggressiveness profile changes (due to calendar rule transition or override).

```json
{
  "type": "profile.change",
  "data": {
    "profile_id": "prof_peak_export",
    "profile_name": "Peak Export",
    "source": "recurring_rule",
    "rule_name": "Weekday evening peak",
    "active_until": "2026-03-25T20:00:00+11:00",
    "timestamp": "2026-03-25T16:00:00+11:00"
  }
}
```

#### `system.alert`

Sent for system-level alerts (API failures, connectivity issues).

```json
{
  "type": "system.alert",
  "data": {
    "severity": "warning",
    "message": "FoxESS API unreachable, using last known state",
    "timestamp": "2026-03-25T10:06:00+11:00"
  }
}
```

### Client -> Server Messages

#### `ping`

Keepalive. Server responds with `pong`.

```json
{ "type": "ping" }
```

## Error Responses

All error responses follow a consistent format:

```json
{
  "error": {
    "code": "INVALID_PARAMETER",
    "message": "min_soc must be between 0 and 100",
    "details": {}
  }
}
```

**HTTP Status Codes:**
| Code | Meaning |
|------|---------|
| 400 | Bad request (invalid parameters) |
| 401 | Unauthorized (missing or invalid API key) |
| 404 | Resource not found |
| 409 | Conflict (e.g., override already active) |
| 500 | Internal server error |
| 503 | Service unavailable (external API down) |

## Type Sharing: Backend -> Frontend

The backend defines all API shapes as **Pydantic v2 models** in `src/api/models/`. FastAPI auto-generates an OpenAPI 3.1 spec from these models at `/openapi.json`.

### Backend (Python -- source of truth)

```python
# src/api/models/battery.py
from pydantic import BaseModel
from typing import Literal

class BatteryState(BaseModel):
    soc: float
    power_w: float
    mode: Literal["charging", "discharging", "holding", "idle"]
    capacity_kwh: float
    min_soc: int
    charge_rate_w: float
    discharge_rate_w: float
    temperature: float | None
    updated_at: str

class ScheduleAction(str, Enum):
    CHARGE = "CHARGE"
    HOLD = "HOLD"
    DISCHARGE = "DISCHARGE"
    AUTO = "AUTO"

class PriceDescriptor(str, Enum):
    SPIKE = "spike"
    HIGH = "high"
    NEUTRAL = "neutral"
    LOW = "low"
    NEGATIVE = "negative"
```

### Frontend (TypeScript -- generated from OpenAPI spec)

```bash
# Generate TypeScript types from running backend
npx openapi-typescript http://localhost:3000/openapi.json -o src/types/api.ts
```

Both the web dashboard (Next.js) and mobile app (React Native) import from the generated types file. This replaces the original `@battery-brain/shared` hand-maintained TypeScript package.

### Key domain types

| Type | Values |
|------|--------|
| BatteryMode | `charging`, `discharging`, `holding`, `idle` |
| ScheduleAction | `CHARGE`, `HOLD`, `DISCHARGE`, `AUTO` |
| PriceDescriptor | `spike`, `high`, `neutral`, `low`, `negative` |
| AlertSeverity | `info`, `warning`, `error` |
| ProfileSource | `default`, `recurring_rule`, `one_off_override` |
